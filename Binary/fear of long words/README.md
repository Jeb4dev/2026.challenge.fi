## Writeup

> Disclaimer: This writeup is copywritten by Gemini 3 (read AI), from the original writeup written by me (see below).

We have a [program](ordbog.c) where a user can add words to a dictionary and show the words in the dictionary.
There is a `win()` function that prints the flag, but it is never actually called in the program.
Our goal is to exploit a buffer overflow bug in the `add` command to overwrite the return address of the function and make it jump directly to `win()`.

The `make_word()` function (which handles the `add` command) allocates a buffer of 64 bytes to store the word, but it uses `fread` to read our input without actually checking if the length we provide is larger than the buffer.

But how can we exploit this?

We can tell the program exactly how long of a word we want to add, even though only 64 bytes are reserved for it. Writing more than 64 bytes will cause a buffer overflow, allowing us to overwrite the return address of the `make_word()` function.

Using GDB, we can find the exact memory address of the `win()` function. Then, we just need to calculate the exact distance (offset) from the start of our input to the function's return address. When `make_word()` finishes and tries to return, it will grab our injected address, jump to `win()`, and print the flag.

Using GDB to analyze the binary:

```text

pwndbg> info addr win

pwndbg> disass *make_word

   0x080492e8 <+4>:     sub    esp,0x54

```

From the disassembly, we can see where our input starts: `lea eax,[ebp-0x4c]`. We also know that the return address is stored just above the base pointer at `[ebp+0x4]`.

So how do we calculate the exact offset needed to overwrite that return address? Here is the full breakdown of the stack frame:

```text
12 bytes of bottom padding (unused space below the buffer to reach 16-byte alignment) +
64 bytes (char buffer[64]) +
4 bytes (char *word pointer) + 
4 bytes padding by compiler (internal alignment) +
4 bytes of Saved EBX (push ebx at the start of the function)
= 88 bytes total space between ESP and EBP.

But since fread starts writing at ebp-0x4c (76 bytes below EBP), our path to the return address is:
  76 bytes (buffer + word + padding + Saved EBX) 
+  4 bytes (Saved EBP - the memory address of the calling function)
= 80 bytes of offset (junk data)

```

After sending 80 bytes of junk data, the next 4 bytes will cleanly overwrite the return address. We will overwrite it with the address of the `win()` function (`0x08049256`) to get the flag.
